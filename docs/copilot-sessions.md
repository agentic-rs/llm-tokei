# Copilot Sessions

GitHub Copilot Chat stores session data under each VS Code workspace storage
directory. `llm-tokei` currently reads two related JSONL artifacts:

- `chatSessions/<session-id>.jsonl`
- `GitHub.copilot-chat/transcripts/<session-id>.jsonl`

These files are related, but they are not interchangeable. `transcripts` are
chronological event logs. `chatSessions` are patch logs that can be replayed
into a final session object.

## File Discovery

For each workspace directory under `workspaceStorage`, Copilot session data can
appear in these locations:

```text
<workspaceStorage>/<workspace-id>/chatSessions/<session-id>.jsonl
<workspaceStorage>/<workspace-id>/GitHub.copilot-chat/transcripts/<session-id>.jsonl
```

`chatSessions` files are useful for estimating per-request usage from the final
session state. `transcripts` files are useful when they contain exact shutdown
usage metrics. When exact transcript metrics exist for a session, they should
take precedence over estimated chat-session records for that same session.

## `transcripts/*.jsonl`

Transcript files are append-style event logs. Each line is a complete JSON
event with a `type`, `data`, `id`, `timestamp`, and usually `parentId`.

Common event shapes:

```json
{
  "type": "assistant.message",
  "data": {
    "messageId": "...",
    "content": "...",
    "toolRequests": [
      {
        "toolCallId": "...",
        "name": "...",
        "arguments": "{\"path\":\"...\"}",
        "type": "function"
      }
    ],
    "reasoningText": ""
  },
  "timestamp": "2026-04-17T17:24:29.659Z"
}
```

```json
{
  "type": "session.shutdown",
  "data": {
    "modelMetrics": {
      "gpt-5-mini": {
        "usage": {
          "inputTokens": 10,
          "outputTokens": 20,
          "cacheReadTokens": 3,
          "cacheWriteTokens": 4
        },
        "requests": { "count": 2 }
      }
    }
  }
}
```

Important transcript fields:

- `type`: event kind, such as `session.start`, `assistant.message`, `tool.execution_start`, `tool.execution_complete`, or `session.shutdown`.
- `timestamp`: event time.
- `data.sessionId`: session id when emitted by `session.start` or shutdown-style events.
- `data.messageId`: assistant message id for `assistant.message`.
- `data.content`: assistant-visible text content for `assistant.message`.
- `data.toolRequests[].arguments`: tool-call arguments generated by the model.
- `data.modelMetrics.*.usage.inputTokens`: exact input tokens when shutdown metrics exist.
- `data.modelMetrics.*.usage.outputTokens`: exact output tokens when shutdown metrics exist.
- `data.modelMetrics.*.usage.cacheReadTokens`: exact cached input tokens when shutdown metrics exist.
- `data.modelMetrics.*.usage.cacheWriteTokens`: exact cache write tokens when shutdown metrics exist.
- `data.modelMetrics.*.requests.count`: request count associated with the model metrics.

Transcript caveats:

- Not every transcript contains `type: "session.shutdown"`.
- `tool.execution_complete` does not always include full tool output in a stable field.
- Message events are good for timelines, but exact billing-style usage should come from `session.shutdown.data.modelMetrics.*.usage` when present.

## `chatSessions/*.jsonl`

Chat session files are patch logs. Each line has a `kind` field:

- `kind: 0`: an initial state snapshot in `v`.
- `kind: 1` or `kind: 2`: a patch with path `k` and value `v`.

Example patch:

```json
{
  "kind": 2,
  "k": ["requests", 0, "response"],
  "v": [
    {
      "kind": "text",
      "value": "Done."
    }
  ]
}
```

To build the final session state, start with the `kind: 0` value and apply
each patch by writing `v` at path `k`. After replay, the main data lives under
top-level fields such as `sessionId`, `customTitle`, `inputState`, and
`requests`.

Important final-state fields:

- `sessionId`: session id.
- `customTitle`: user-visible session title when present.
- `creationDate`: session creation timestamp in milliseconds.
- `inputState.selectedModel.metadata.family`: selected model family fallback.
- `inputState.selectedModel.metadata.id`: selected model id fallback.
- `requests[].requestId`: request id.
- `requests[].timestamp`: request timestamp in milliseconds.
- `requests[].modelId`: request model id.
- `requests[].agent.modelId`: alternate request model id.
- `requests[].response`: rendered response items.
- `requests[].completionTokens`: exact output tokens when present.
- `requests[].result.metadata.renderedUserMessage`: rendered user prompt content.
- `requests[].result.metadata.renderedGlobalContext`: rendered context content.
- `requests[].result.metadata.toolCallRounds`: internal tool-call continuation rounds.
- `requests[].result.metadata.toolCallRounds[].thinking.tokens`: exact reasoning tokens when present.
- `requests[].result.metadata.toolCallRounds[].response`: tool round response text. This can be a short summary rather than the full tool payload.
- `requests[].result.metadata.toolCallRounds[].toolCalls[].id`: tool-call id, used to match `toolCallResults` entries.
- `requests[].result.metadata.toolCallRounds[].toolCalls[].arguments`: tool-call arguments generated by the model.
- `requests[].result.metadata.toolCallResults`: map of tool-call id to serialized tool result payload.
- `requests[].modeInfo.modeId`: chat mode.
- `requests[].agent.id`: agent id.
- `requests[].command` and `requests[].slashCommand.command`: command metadata, including compaction detection.

Response item fields worth recognizing:

- `requests[].response[].kind`: response item kind.
- `requests[].response[].value`: rendered text for `kind: "text"` and some markdown-like items.
- `requests[].response[].invocationMessage.value`: user-visible tool invocation text for `kind: "toolInvocationSerialized"`.
- `requests[].response[].resultDetails.input`: tool input command or payload in some tool invocation records.
- `requests[].response[].resultDetails.output`: tool output payload in some tool invocation records.
- `requests[].response[].toolSpecificData.terminalCommandOutput.text`: terminal command output in some records.

Tool call result fields worth recognizing:

- `requests[].result.metadata.toolCallResults`: object keyed by the full tool call id, such as `call_...__vscode-...`.
- `requests[].result.metadata.toolCallResults[toolCallId].$mid`: serialized VS Code marker for the result wrapper.
- `requests[].result.metadata.toolCallResults[toolCallId].content`: array of serialized result content items.
- `requests[].result.metadata.toolCallResults[toolCallId].content[].$mid`: serialized content marker. In observed samples, `$mid: 21` stores plain string output and `$mid: 23` stores rich tree output.
- `requests[].result.metadata.toolCallResults[toolCallId].content[].value`: result content. This can be a plain string, or an object with a `node` rich-text tree.
- `requests[].result.metadata.toolCallResults[toolCallId].content[].value.node`: rich-text root for structured output, with `type`, `ctor`, `ctorName`, `children`, `props`, and `references`.
- `requests[].result.metadata.toolCallResults[toolCallId].content[].value.node.children[]`: nested rich-text nodes. Leaf text usually appears on child nodes as `text`, sometimes with `references` anchors for file paths or URIs.

`toolCallResults` entries are keyed by the same ids emitted in
`toolCallRounds[].toolCalls[].id`. They can contain full terminal output, file
read output, patch results, search results, and other tool payloads. Prefer
these payloads for estimated tool-result input when they are available.
`toolCallRounds[].response` is useful as a fallback, but observed sessions often
store only a one-line summary there.

## State View vs Raw View

There are two ways to analyze `chatSessions`.

`state` view means replaying patches and inspecting the final reconstructed
session object. This is what `llm-tokei` uses for Copilot chat-session
estimates, because it gives one stable record per final request.

`raw` view means reading every JSONL line directly and counting each patch value
as it appears. This is useful for auditing byte volume, especially terminal
outputs. A later patch can replace `k: ["requests", 0, "response"]`, so a large
intermediate `requests[].response[].resultDetails.output` value may exist in
the raw patch stream but not survive in final state.

Use `state` when the question is “what did the final session contain?” Use
`raw` when the question is “what payloads were emitted over time?”

## Usage Estimation Rules

Copilot chat-session files do not reliably persist per-turn input/output token
counts. When exact `session.shutdown` metrics are unavailable, `llm-tokei`
estimates token counts from character counts using roughly four characters per
token.

Input estimate includes:

- `requests[].result.metadata.renderedUserMessage`
- `requests[].result.metadata.renderedGlobalContext`
- `requests[].result.metadata.toolCallResults` content matched by `toolCallRounds[].toolCalls[].id`
- `requests[].result.metadata.toolCallRounds[].response` only when matching `toolCallResults` content is unavailable

Output estimate includes:

- text-like `requests[].response[]` items
- `requests[].result.metadata.toolCallRounds[].toolCalls[].arguments`

Reasoning uses exact emitted values when present:

- `requests[].result.metadata.toolCallRounds[].thinking.tokens`

Output can be exact for a request when this field exists:

- `requests[].completionTokens`

Tool-call accounting follows the model perspective:

- Tool-call arguments are model output because the model generated them.
- Tool-call results are model input because they are fed back into later model context.
- Prefer `toolCallResults` for estimated model input; use `toolCallRounds[].response` as a fallback because it can be a short summary.

## Deduplication

A single Copilot session may have both `chatSessions/<session-id>.jsonl` and
`transcripts/<session-id>.jsonl`.

When transcript shutdown metrics exist, they are treated as exact records. The
corresponding estimated chat-session records for that same `sessionId` should be
dropped to avoid double counting.

When shutdown metrics do not exist, the chat-session estimate remains useful and
should be marked as estimated.

## Common Count Mismatches

Count mismatches usually come from comparing different layers:

- `transcripts[].data.content` is assistant message event content.
- `chatSessions` final `requests[].response` is final rendered UI state.
- `chatSessions` raw patches include intermediate values that final state may overwrite.
- `requests[].result.metadata.toolCallRounds[].response` may be a summary of tool output, not the full assistant-visible or model-visible payload.
- `requests[].result.metadata.toolCallResults` is keyed raw result payload data and may overlap with `toolCallRounds[].response`; avoid counting both for the same tool call.
- `requests[].response[].toolSpecificData.terminalCommandOutput.text` may include ANSI-colored terminal output that differs from plain text elsewhere.

For billing-style totals, prefer exact `session.shutdown.data.modelMetrics.*.usage`
when available. For estimated totals, use final `chatSessions` state and mark
the result as approximate. For byte-volume audits, inspect raw `chatSessions`
patches.
